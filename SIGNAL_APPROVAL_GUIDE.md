## Overview

The Signal Approval workflow intercepts auto-generated signals before they reach the trading engine. Every signal generated by the Hybrid Portfolio System v3.0 is stamped `PENDING` and held for manager review. Managers can:

- **Approve** — send the signal to trading as-is
- **Reject** — discard the signal with a mandatory reason
- **Adjust** — modify entry price, TP levels, and/or SL, then automatically approve

All decisions are immutably logged in the `signal_approval_audit` collection.

## Signal Lifecycle

```
Auto-generated signal
        │
        ▼
   [PENDING] ◄─── Awaiting manager review
        │
   ┌────┴────┐
   │         │
   ▼         ▼
[APPROVED] [REJECTED]
   │    ▲
   │    └── [ADJUSTED] (modified then approved)
   ▼
Trading Engine
```

| Status | Description | Trading |
|---|---|---|
| `PENDING` | Awaiting manager review | Held — not traded |
| `APPROVED` | Approved as-is | Active — sent to trading |
| `REJECTED` | Rejected by manager | Cancelled — not traded |
| `ADJUSTED` | Modified and approved | Active — traded with new levels |

## API Reference

All endpoints are prefixed with `/api/signals` and require:
```
Authorization: Bearer <manager_jwt>
```

### GET /api/signals/pending
List all signals awaiting manager review.

**Query Parameters:**
| Parameter | Type | Default | Description |
|---|---|---|---|
| `pair` | string | — | Filter by pair (e.g. `XAUUSD`) |
| `limit` | integer | 50 | Max results (1–200) |
| `min_confidence` | float | — | Minimum confidence % |

**Response:**
```json
{
  "success": true,
  "signals": [...],
  "count": 3,
  "status": "PENDING"
}
```

---

### GET /api/signals/{signal_id}
Get full details for a single signal, including all analysis fields and approval history.

**Response includes:**
- Entry price, TP levels, SL
- Confidence score
- Regime, SMC score, MTF alignment, pivot zone
- Technical indicators (RSI, MACD, MA20/50, ATR)
- Complete approval audit trail

---

### POST /api/signals/{signal_id}/approve
Approve a pending signal.

**Request Body:**
```json
{
  "notes": "Strong SMC confluence at key support. Approved."
}
```

**Response:**
```json
{
  "success": true,
  "signal_id": "abc123",
  "approval_status": "APPROVED",
  "approved_by": "mgr-uuid",
  "approved_at": "2025-01-15T10:30:00",
  "message": "Signal approved and queued for trading"
}
```

---

### POST /api/signals/{signal_id}/reject
Reject a pending signal. A reason is mandatory (minimum 5 characters).

**Request Body:**
```json
{
  "reason": "High-impact NFP news in 20 minutes. Too risky."
}
```

**Response:**
```json
{
  "success": true,
  "signal_id": "abc123",
  "approval_status": "REJECTED",
  "rejected_by": "mgr-uuid",
  "rejected_at": "2025-01-15T10:31:00",
  "rejection_reason": "High-impact NFP news in 20 minutes. Too risky.",
  "message": "Signal rejected and removed from trading queue"
}
```

---

### POST /api/signals/{signal_id}/adjust
Adjust entry price, TP levels, and/or SL, then automatically approve.

At least one of `entry_price`, `tp_levels`, or `sl_price` must be provided.

**Request Body:**
```json
{
  "entry_price": 2645.50,
  "tp_levels": [2658.00, 2672.00, 2690.00],
  "sl_price": 2632.00,
  "adjustment_notes": "Tightened entry to key support zone. Extended TP3 to weekly resistance."
}
```

**Response:**
```json
{
  "success": true,
  "signal_id": "abc123",
  "approval_status": "ADJUSTED",
  "adjusted_by": "mgr-uuid",
  "adjusted_at": "2025-01-15T10:32:00",
  "changes": {
    "entry_price": {"from": 2648.20, "to": 2645.50},
    "tp_levels":   {"from": [2660.00, 2675.00, 2685.00], "to": [2658.00, 2672.00, 2690.00]},
    "sl_price":    {"from": 2630.00, "to": 2632.00},
    "risk_reward": {"from": 1.8, "to": 2.1}
  },
  "new_risk_reward": 2.1,
  "message": "Signal adjusted and approved for trading"
}
```

## Adjustment Guidelines

### When to Adjust Entry Price

- The auto-generated entry is slightly above/below a key structural level (order block, FVG, pivot)
- You want to improve the R:R by entering closer to a support/resistance zone
- The entry is valid but the current price has moved slightly since generation

**Gold (XAUUSD/XAUEUR) entry adjustment rules:**
- Keep entry within ±$5.00 of the original for intraday signals
- Keep entry within ±$15.00 for swing signals
- Never adjust entry past the nearest TP level

### When to Adjust TP Levels

- TP1 is too close to entry (less than 1× ATR away)
- TP3 is beyond a major structural resistance/support level
- You want to align TPs with pivot points, order blocks, or weekly levels
- Extending TP3 to capture a larger move when trend is strong

**TP adjustment rules:**
- All TP levels must be in the correct direction (BUY: above entry; SELL: below entry)
- TP levels should be ordered from nearest to furthest
- Minimum TP1 distance: 1× ATR from entry (approximately $8–12 for Gold)
- Maximum TP3 distance: 5× ATR from entry (approximately $40–60 for Gold)

### When to Adjust Stop Loss

- The auto-generated SL is inside a liquidity zone (likely to be swept)
- The SL is too tight relative to current volatility (ATR)
- The SL is too wide, reducing R:R below 1.5

**SL adjustment rules:**
- BUY signals: SL must be below entry
- SELL signals: SL must be above entry
- Minimum SL distance: 0.5× ATR from entry (approximately $4–6 for Gold)
- Maximum SL distance: 3× ATR from entry (approximately $24–36 for Gold)
- Place SL below/above swing lows/highs, not at round numbers

### R:R Validation

The system automatically recalculates R:R after any adjustment using TP1 as the target:

```
R:R = |TP1 - Entry| / |Entry - SL|
```

Minimum acceptable R:R after adjustment: **1.5** (system enforces this via geometry validation).

## Best Practices

### Signal Review Checklist

Before approving any signal, verify:

1. **Regime alignment** — Is the signal direction consistent with the current market regime?
   - TREND_UP → prefer BUY signals
   - TREND_DOWN → prefer SELL signals
   - RANGE → prefer mean-reversion signals at extremes

2. **SMC score** — Aim for SMC score ≥ 6/10 for live trading

3. **MTF alignment** — Prefer signals with MTF alignment ≥ 60%

4. **Confidence** — Minimum 65% for approval; prefer 70%+ for full position size

5. **News calendar** — Check for high-impact events within 30 minutes
   - USD news: affects XAUUSD directly
   - EUR news: affects XAUEUR directly
   - Avoid trading 30 minutes before/after FOMC, NFP, CPI releases

6. **Session timing** — Prefer London (07:00–16:00 UTC) and NY (13:00–21:00 UTC) sessions

7. **Price structure** — Confirm entry is at a valid structural level (support/resistance, order block, FVG)

### Rejection Best Practices

Always provide a specific, actionable rejection reason:

| ❌ Poor reason | ✅ Good reason |
|---|---|
| "Bad signal" | "Confidence 58% below 65% threshold" |
| "Not good" | "FOMC in 20 minutes — news risk too high" |
| "Wrong direction" | "TREND_DOWN regime — BUY signal against trend" |
| "Too risky" | "SL only 3 pips from entry — insufficient buffer" |

### Adjustment Best Practices

- Always add `adjustment_notes` explaining the rationale
- Make minimal adjustments — only change what is necessary
- Document the structural reason (e.g. "SL moved below swing low at 2631.50")
- If more than 2 levels need adjustment, consider rejecting and waiting for a cleaner setup

## Approval Statistics and Reporting

### Key Metrics to Monitor

| Metric | Target | Action if Outside Target |
|---|---|---|
| Approval rate | 60–80% | Review signal generation quality |
| Rejection rate | 20–40% | Investigate if too high (>50%) |
| Avg approved confidence | ≥ 70% | Raise `MIN_CONFIDENCE` env var if consistently low |
| Pending signals | < 10 | Review more frequently if queue builds up |

### Weekly Review Workflow

```bash
# Monday morning review — last 7 days
curl -X GET "https://your-service.railway.app/api/signals/stats?hours=168" \
  -H "Authorization: Bearer $TOKEN"

# Check which pairs had the most rejections
# Look at per_pair breakdown in the response

# Review all adjusted signals to validate adjustment quality
curl -X GET "https://your-service.railway.app/api/signals/history?status=ADJUSTED&hours=168" \
  -H "Authorization: Bearer $TOKEN"
```

### Per-Manager Performance

Track individual manager approval patterns to ensure consistency:

```bash
# Stats for a specific manager over 30 days
curl -X GET "https://your-service.railway.app/api/signals/stats?hours=720&manager_id=MGR_UUID" \
  -H "Authorization: Bearer $TOKEN"
```

A healthy manager profile shows:
- Consistent approval rate (not wildly varying week to week)
- Rejection reasons that are specific and categorisable
- Adjustments that improve R:R (visible in `changes.risk_reward`)

## Audit Trail

Every signal action is permanently recorded in the `signal_approval_audit` MongoDB collection. Records are immutable and include:

| Field | Description |
|---|---|
| `audit_id` | Unique UUID for the audit record |
| `timestamp` | UTC timestamp of the action |
| `action` | `signal:approve`, `signal:reject`, `signal:adjust`, `signal:view_details`, `signal:list_pending` |
| `performed_by` | Manager ID who performed the action |
| `role` | Manager role at time of action (ADMIN/MANAGER/VIEWER) |
| `signal_id` | ID of the affected signal |
| `details` | Action-specific details (pair, type, entry, reason, changes) |
| `success` | Whether the action succeeded |
| `error` | Error message if action failed |

### Querying the Audit Trail

Use the existing audit log endpoint to filter signal actions:

```bash
# All signal approval actions in the last 24 hours
curl -X GET "https://your-service.railway.app/api/manager/audit?action=signal:approve&since_hours=24" \
  -H "Authorization: Bearer $TOKEN"

# All signal rejections by a specific manager
curl -X GET "https://your-service.railway.app/api/manager/audit?action=signal:reject&manager_id=MGR_UUID" \
  -H "Authorization: Bearer $TOKEN"

# All signal adjustments
curl -X GET "https://your-service.railway.app/api/manager/audit?action=signal:adjust" \
  -H "Authorization: Bearer $TOKEN"
```
